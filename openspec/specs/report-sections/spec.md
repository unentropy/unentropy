## Overview

Report sections allow users to organize metrics into named, visually distinct groups within the HTML report. This capability transforms the flat report layout into a structured dashboard where related metrics (e.g., size, tests, refactoring progress) appear together under section headers.

## ADDED Requirements

### Requirement: Section Configuration

As a user, I want to define named sections in my configuration so that related metrics are grouped together in the report.

The system SHALL support configurable report sections in the report configuration, where each section contains a name, optional description, and list of charts. Charts within a section SHALL appear in their definition order from the configuration file.

#### Scenario: Sections defined in configuration

- **GIVEN** a report configuration with two sections: "Code Size" and "Test Coverage"
- **WHEN** the report is generated
- **THEN** the report displays "Code Size" as a section header with its charts underneath
- **THEN** the report displays "Test Coverage" as a separate section header with its charts underneath
- **THEN** charts within each section appear in the same order as defined in the configuration

#### Scenario: Section with description

- **GIVEN** a section configuration includes a description "Metrics tracking bundle and source size"
- **WHEN** the report is generated
- **THEN** the section header displays the name and the description as subtitle text

#### Scenario: Empty section list

- **GIVEN** a report configuration defines an empty sections array
- **WHEN** the report is generated
- **THEN** the report falls back to the default flat layout without section headers

#### Scenario: Unreferenced metrics

- **GIVEN** five metrics are defined in the configuration
- **GIVEN** only three of those metrics are referenced in report sections
- **WHEN** the report is generated
- **THEN** the three referenced metrics appear in their assigned sections
- **THEN** the two unreferenced metrics are omitted from the report with no warning

## MODIFIED Requirements

### Requirement: Report Structure

The system SHALL generate a self-contained HTML report with a header, metric sections in a responsive grid, summary statistics per metric, and a footer. When a report configuration with sections is present, charts SHALL be rendered within their assigned sections instead of a flat grid.

#### Scenario: Full report with multiple metrics and sections

- **GIVEN** a database with 6 metrics tracked over 15 builds
- **GIVEN** a report configuration with 2 sections containing 3 charts each
- **WHEN** the report is generated and opened
- **THEN** the header displays the repository name, generation timestamp, date range of data, and total build count
- **THEN** each section displays its name as a header with optional description
- **THEN** charts within each section are displayed in a responsive grid layout in their definition order
- **THEN** each chart card includes the metric name, optional description, and a visual chart
- **THEN** each numeric chart card displays summary statistics: Latest, Min, Max, and Trend (direction + percentage)
- **THEN** the footer displays "Generated by Unentropy" with version and a documentation link

#### Scenario: Full report without section configuration

- **GIVEN** a database with 3 metrics tracked over 15 builds
- **GIVEN** no report configuration is present
- **WHEN** the report is generated and opened
- **THEN** all charts are displayed in a single flat responsive grid without section headers

## Key Entities

- **Section**: A named grouping of charts in the report with an optional description.
- **Report Configuration**: Optional advanced configuration block in `unentropy.json` controlling report layout and behavior.

## Related

- `reporting` — Sections modify how the report structure is rendered.
- `multi-metric-charts` — Sections can contain both single-metric and multi-metric charts.
