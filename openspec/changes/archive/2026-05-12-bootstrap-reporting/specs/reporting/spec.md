## ADDED Requirements

### Requirement: Report Structure
The system SHALL generate a self-contained HTML report with a header, metric sections in a responsive grid, summary statistics per metric, and a footer.

#### Scenario: Full report with multiple metrics
- **GIVEN** a database with 3 metrics tracked over 15 builds
- **WHEN** the report is generated and opened
- **THEN** the header displays the repository name, generation timestamp, date range of data, and total build count
- **THEN** each metric is displayed in its own card within a responsive grid layout
- **THEN** each metric card includes the metric name, optional description, and a visual chart
- **THEN** each numeric metric card displays summary statistics: Latest, Min, Max, and Trend (direction + percentage)
- **THEN** the footer displays "Generated by Unentropy" with version and a documentation link

#### Scenario: Empty state with no metrics
- **GIVEN** a database with no metrics collected
- **WHEN** the report is generated
- **THEN** the report displays a friendly message with guidance on how to start collecting data

#### Scenario: Sparse data warning
- **GIVEN** a metric has fewer than 5 data points
- **WHEN** the report is generated
- **THEN** a yellow warning banner displays indicating limited data availability

---

### Requirement: Chart Types
The system SHALL render numeric metrics as line charts with smooth curves and filled area, and label metrics as bar charts showing occurrence counts.

#### Scenario: Numeric metric line chart
- **GIVEN** a metric with numeric values
- **WHEN** the report is generated
- **THEN** the metric is displayed as a line chart with a smooth curve and filled area under the line
- **THEN** axes are clearly labeled (X-axis: build date, Y-axis: metric value)

#### Scenario: Label metric bar chart
- **GIVEN** a metric with label values
- **WHEN** the report is generated
- **THEN** the metric is displayed as a bar chart showing occurrence counts per label

#### Scenario: Single data point
- **GIVEN** a metric with data for only 1 build
- **WHEN** the report is generated
- **THEN** the single point is displayed as a dot on the chart
- **THEN** summary statistics show that value as Latest/Min/Max
- **THEN** trend displays "N/A"

#### Scenario: Flatline data
- **GIVEN** all metric values are identical
- **WHEN** the chart is rendered
- **THEN** the Y-axis shows a reasonable scale
- **THEN** the trend indicator shows "stable" (→)

---

### Requirement: Interactive Tooltips
Charts SHALL display interactive tooltips on hover showing exact value, formatted timestamp, and commit SHA (first 7 characters).

#### Scenario: Tooltip on hover
- **GIVEN** a chart with data points
- **WHEN** the user hovers over a data point
- **THEN** a tooltip appears showing the exact value, formatted timestamp, and commit SHA (first 7 characters)

#### Scenario: Tooltip for missing data
- **GIVEN** a build with no recorded value for a metric
- **WHEN** the user hovers over that build position
- **THEN** the tooltip indicates "No data for this build"

---

### Requirement: Normalized X-Axis
All charts SHALL display the complete build history on the X-axis spanning from the first to the last build in the database, with visual gaps for missing data rather than interpolation.

#### Scenario: Consistent X-axis across metrics
- **GIVEN** metrics with varying data availability across builds
- **WHEN** the user views the report
- **THEN** all charts display the same X-axis range covering all builds in the database

#### Scenario: Gap for missing data
- **GIVEN** a metric missing a value for a specific build
- **WHEN** the chart is rendered
- **THEN** the missing data point is visually indicated as a gap in the line (not interpolated or connected)

#### Scenario: Independent Y-axis scaling
- **GIVEN** a metric with large values and another with small values
- **WHEN** both charts are rendered
- **THEN** each chart maintains its own Y-axis scale optimized for its data range

---

### Requirement: Preview Data Toggle
When total build count is less than 10, the report SHALL display a preview bar with a toggle to switch between real data and synthetic demonstration data.

#### Scenario: Preview bar visible for sparse data
- **GIVEN** a report with fewer than 10 total builds
- **WHEN** the user opens the report
- **THEN** a toggle switch appears in the header area with the label "Show preview" in the ON position by default

#### Scenario: Toggle to real data
- **GIVEN** the toggle is visible and in the ON position
- **WHEN** the user clicks the toggle to OFF
- **THEN** all charts switch to displaying actual recorded data

#### Scenario: Toggle to synthetic data
- **GIVEN** the toggle is in the OFF position showing real sparse data
- **WHEN** the user clicks the toggle to ON
- **THEN** all charts display synthetic data with 20 data points spanning 60 days with a plausible trend

#### Scenario: Toggle hidden for sufficient data
- **GIVEN** a report with 10 or more total builds
- **WHEN** the user opens the report
- **THEN** no toggle is displayed and only real data is shown

#### Scenario: Preview with zero builds
- **GIVEN** a report with 0 builds but configured metrics
- **WHEN** the user opens the report
- **THEN** the toggle appears ON by default showing synthetic data
- **THEN** toggling OFF shows empty charts

---

### Requirement: Synchronized Crosshair
When hovering over any chart, a vertical alignment line SHALL appear on all visible charts at the same X-axis position, and all charts SHALL display tooltips for the same build with synchronization latency under 50ms.

#### Scenario: Vertical line across all charts
- **GIVEN** a report with multiple metrics
- **WHEN** the user hovers over a data point on Chart A
- **THEN** a vertical alignment line appears on all visible charts at the same X-axis position
- **THEN** all charts display tooltips showing their values for the same build

#### Scenario: Real-time sync on move
- **GIVEN** the alignment line and tooltips are displayed
- **WHEN** the user moves the mouse to a different X position
- **THEN** the vertical line updates in real-time on all charts
- **THEN** all tooltips simultaneously reflect the new build data
- **THEN** synchronization latency is less than 50ms across all charts

#### Scenario: Missing data in sync
- **GIVEN** a metric with no data for a specific build
- **WHEN** the user hovers over that build on another chart
- **THEN** the sparse metric's chart shows the vertical line at the correct X position
- **THEN** the tooltip indicates "No data for this build"

#### Scenario: Dismiss on leave
- **GIVEN** tooltips and alignment line are displayed
- **WHEN** the user moves the mouse away from all charts
- **THEN** the vertical alignment line and all tooltips are simultaneously dismissed

---

### Requirement: Drag-to-Zoom
Charts SHALL support drag-to-zoom by clicking and dragging horizontally to select a range, with synchronization across all charts and a minimum selection of 4 data points.

#### Scenario: Drag to select zoom range
- **GIVEN** a chart displaying data with 10+ data points
- **WHEN** the user clicks and drags horizontally on a chart
- **THEN** a semi-transparent selection box appears showing the zoom range

#### Scenario: Zoom on release
- **GIVEN** a selection box is visible
- **WHEN** the user releases the mouse button
- **THEN** the chart zooms to show only the selected range (minimum 4 data points)

#### Scenario: Zoom disabled for few data points
- **GIVEN** a chart with fewer than 10 data points
- **WHEN** the user tries to drag-select
- **THEN** zoom is disabled and no selection box appears

---

### Requirement: Preset Date Filters
The report SHALL display filter buttons ("7 days", "30 days", "90 days", "All", "Custom") that update all charts to display only data within the selected time range.

#### Scenario: Filter buttons visible
- **GIVEN** a report is opened
- **WHEN** the user views the header area
- **THEN** they see filter buttons: "7 days", "30 days", "90 days", "All", "Custom"

#### Scenario: Apply preset filter
- **GIVEN** the filter buttons are visible
- **WHEN** the user clicks "30 days"
- **THEN** all charts update to show only data from the last 30 days

#### Scenario: Active filter highlighted
- **GIVEN** a filter is active
- **WHEN** the user views the filter buttons
- **THEN** the active filter is visually highlighted

#### Scenario: Default filter
- **GIVEN** a report is first opened
- **WHEN** no filter has been selected
- **THEN** "All" is the default active filter

#### Scenario: Empty date range
- **GIVEN** a date range filter is applied
- **WHEN** no data exists within that range
- **THEN** each chart displays an empty state message within the chart area

---

### Requirement: Custom Date Range Filter
The system SHALL provide a custom date picker popover with "From" and "To" inputs, constraint validation, and integration with drag-to-zoom.

#### Scenario: Open custom popover
- **GIVEN** the user clicks the "Custom" button
- **WHEN** the popover opens
- **THEN** they see "From" and "To" date pickers with calendar dropdowns and a "Clear" button

#### Scenario: Apply custom range
- **GIVEN** the custom date picker popover is open
- **WHEN** the user selects valid "From" and "To" dates
- **THEN** all charts immediately filter to that date range
- **THEN** the "Custom" button is visually highlighted as active

#### Scenario: Invalid date range
- **GIVEN** the custom date picker is open
- **WHEN** the user selects a "From" date that is after the "To" date
- **THEN** an error message is displayed and the filter is not applied

#### Scenario: Date constraints enforced
- **GIVEN** the calendar picker is displayed
- **WHEN** the user views dates
- **THEN** dates before the earliest data point and after the latest data point are grayed out and not selectable

#### Scenario: Custom range cleared by preset
- **GIVEN** a custom date range is active
- **WHEN** the user clicks a preset filter
- **THEN** the custom range is cleared and the preset filter is applied

#### Scenario: Clear resets to All
- **GIVEN** a custom date range is active
- **WHEN** the user clicks "Clear" in the popover
- **THEN** the filter resets to "All" and the popover closes

#### Scenario: Zoom activates custom filter
- **GIVEN** the user drags to zoom on any chart
- **WHEN** the zoom is applied
- **THEN** the "Custom" button becomes active with the zoomed date range

#### Scenario: Popover close on outside click
- **GIVEN** the custom date picker popover is open
- **WHEN** the user clicks outside the popover
- **THEN** the popover closes while preserving any valid custom range

#### Scenario: Empty custom range
- **GIVEN** a custom date range contains no data points
- **WHEN** the filter is applied
- **THEN** each chart displays "No data in selected range"

### Requirement: Chart PNG Export
Each chart card SHALL display a "Download PNG" button that exports the chart as a PNG image, including metric name as title and a preview watermark when preview data is active.

#### Scenario: Export button visible
- **GIVEN** a chart is displayed
- **WHEN** the user views the chart card
- **THEN** they see a "Download PNG" button

#### Scenario: Export current view
- **GIVEN** the user clicks "Download PNG"
- **WHEN** the export completes
- **THEN** a PNG file is downloaded containing the chart with its title
- **THEN** the exported image reflects the current zoom level and date range filter

#### Scenario: Preview watermark
- **GIVEN** the dummy data toggle is active
- **WHEN** the user exports a chart
- **THEN** the exported image includes a "(Preview Data)" watermark

### Requirement: Responsive Design
The report SHALL render correctly on mobile (320px+), tablet (640px+), and desktop (1024px+) screens without horizontal scrolling.

#### Scenario: Mobile layout
- **GIVEN** the report is viewed on a mobile device (320px - 640px)
- **WHEN** the page renders
- **THEN** the header stacks vertically
- **THEN** metric cards display in a single column
- **THEN** summary statistics display in a 2x2 grid
- **THEN** touch targets are at least 44x44px
- **THEN** there is no horizontal scrolling

#### Scenario: Tablet layout
- **GIVEN** the report is viewed on a tablet (640px - 1024px)
- **WHEN** the page renders
- **THEN** the header displays side-by-side
- **THEN** metric cards display in 1-2 columns

#### Scenario: Desktop layout
- **GIVEN** the report is viewed on a desktop (1024px+)
- **WHEN** the page renders
- **THEN** metric cards display in a 2-column grid
- **THEN** all content fits within a max-width container (1280px)

### Requirement: Dark Mode
The report SHALL support both light and dark modes based on user's system preference via `prefers-color-scheme`.

#### Scenario: Dark mode rendering
- **GIVEN** the user's system has dark mode enabled
- **WHEN** the report is opened
- **THEN** all text is readable on dark backgrounds
- **THEN** chart colors are adjusted for dark background
- **THEN** card backgrounds invert appropriately
- **THEN** there are no white or light flashes

### Requirement: Accessibility
The report SHALL meet WCAG 2.1 AA accessibility standards for color contrast, keyboard navigation, and screen reader support.

#### Scenario: Color contrast
- **GIVEN** the report is rendered
- **WHEN** text is displayed
- **THEN** text on background meets minimum 4.5:1 contrast ratio
- **THEN** information is not conveyed by color alone (arrows supplement trend colors)

#### Scenario: Keyboard navigation
- **GIVEN** the report is opened
- **WHEN** a user navigates using the keyboard
- **THEN** all interactive elements are keyboard-accessible
- **THEN** focus indicators are clearly visible
- **THEN** the tab order is logical
- **THEN** the Escape key closes the custom date picker popover

#### Scenario: Screen reader support
- **GIVEN** the report is opened in a screen reader
- **WHEN** charts are rendered
- **THEN** charts have descriptive ARIA labels
- **THEN** summary statistics are readable without viewing charts
- **THEN** filter buttons announce active state
- **THEN** the toggle announces its state

### Requirement: Empty State Handling
The report SHALL gracefully handle all empty or edge case data scenarios with appropriate messaging.

#### Scenario: No metrics collected
- **GIVEN** no metrics have been defined or collected
- **WHEN** the report is generated
- **THEN** a friendly empty state message is displayed with guidance on how to start tracking

#### Scenario: Metric with missing data for a build
- **GIVEN** a metric has no value recorded for a specific build
- **WHEN** the chart is rendered
- **THEN** a gap appears at that position in the line
- **THEN** hovering over that gap shows "No data for this build"

#### Scenario: All data within range
- **GIVEN** a date filter is applied
- **WHEN** the selected range contains no data
- **THEN** charts display "No data in selected range"
- **THEN** statistics show "N/A"
- **THEN** the filter remains active

## Edge Cases

- What happens when the database has zero builds? Display an empty state with guidance message, no toggle shown.
- What happens when a metric has data for only 1 build? Display the single point as a dot on the chart with statistics showing that value as Latest/Min/Max; trend shows "N/A".
- What happens when toggling dummy data while interacting with a chart tooltip? Tooltip dismisses and chart transitions to the new data.
- What happens when all metric values are identical (flatline)? Chart Y-axis shows a reasonable scale; trend indicator shows "stable" (→).
- What happens when a metric has extremely large values vs another with small values? Each chart maintains its own Y-axis scale optimized for its data range.
- What happens when zooming on a chart with fewer than 3 data points? Zoom functionality is disabled.
- What happens when a date range filter is applied but no data exists within that range? Each chart displays "No data in selected range".
- What happens when exporting a chart while dummy data is displayed? The exported PNG includes a "(Preview Data)" watermark.
- What happens when synchronized tooltips are triggered on a chart with no data for that build? The vertical alignment line displays at the corresponding X-axis position and the tooltip displays "No data for this build".
- What happens when hovering over a zoomed-in section of a chart? The vertical alignment line appears within the current chart area bounds.
- What happens when hovering over a bar chart (label metric)? The vertical alignment line appears if the chart has a numeric X-axis (time-based).
- What happens when the user zooms while a date range filter is active? Zoom operates within the filtered range; reset zoom returns to the filtered view.
- What happens when the custom date picker popover is open and the user clicks a preset filter button? The popover immediately closes and the preset filter is applied.
- What happens when CDN resources (Tailwind CSS, Chart.js) fail to load? A graceful fallback message is shown.
- What happens when metric names contain special characters or XSS payloads? All user-provided data is sanitized (HTML escaped) before insertion.
- What happens when viewing the report in print mode? Print stylesheet loads, interactive elements are hidden, page breaks avoid splitting charts.

## Assumptions

- The report is a self-contained HTML file viewable in any modern browser.
- CDN resources (Tailwind CSS, Chart.js, chartjs-adapter-date-fns) are available with graceful fallback.
- Synthetic data generation uses deterministic patterns based on current metric configuration.
- Synthetic data spans 60 days from the current date, regardless of actual data range.
- The preview toggle persists only for the current session (page reload resets to real data).
- Build contexts are ordered chronologically by timestamp for X-axis display.
- All interactive features are rendered client-side using embedded JSON data.
- Zoom synchronization applies to all visible charts (no independent zoom per chart).
- The footer remains static showing total database stats regardless of active filters.

## Dependencies

- Chart.js v4.x via CDN (jsDelivr) for interactive visualizations
- chartjs-adapter-date-fns via CDN for time-axis support
- Tailwind CSS v3.x via CDN for responsive styling
- Modern browser support (Chrome 90+, Firefox 88+, Safari 14+, Edge 90+)
- SQLite database with collected metric data
- Metric configuration from `unentropy.json`

## Scope Boundaries

### In Scope

- Self-contained HTML report generation with embedded JSON data
- Line charts for numeric metrics and bar charts for label metrics
- Interactive tooltips with value, timestamp, and commit SHA
- Normalized X-axis across all charts with gaps for missing data
- Preview data toggle for reports with fewer than 10 builds
- Synchronized crosshair alignment line across all charts
- Drag-to-zoom with minimum 4 data points
- Preset date filters (7d/30d/90d/All)
- Custom date picker with validation
- Chart PNG export with preview watermark
- Responsive design (mobile/tablet/desktop)
- Dark mode support via `prefers-color-scheme`
- WCAG 2.1 AA accessibility
- Print stylesheet support

### Out of Scope

- Storage backend behavior (see `storage/` spec)
- Quality gate evaluation (see `quality-gates/` spec)
- Metric configuration and collection (see `metrics/` spec)
- GitHub Actions workflow orchestration (see `actions/` spec)
- CLI report generation command (see `cli/` spec)
- Multi-repository aggregation
- Advanced analytics (anomaly detection, trend forecasting)
- Custom visualization plugins
- User-customizable chart colors or themes
- Real-time metric streaming
